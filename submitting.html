    Soumission d'application Spark¶

    Une application Spark est coordonnée par l'objet SparkContext créé dans le programme principal appelé Driver. Ce dernier se connecte à un gestionnaire de cluster (Spark, Yarn, Mesos, Kubernetes) qui lui allouera les ressources nécessaires. Le Driver obtient alors les Executors (processus sur les nœuds du cluster) et par la suite leur envoie le code puis les tâches à réaliser. (voir figure ci-après)

    Exécution interactive¶

    C'est une exécution sur le nœud local dont l'utilité est de tester l'application Spark.

    Avec le shell pyspark¶

    Spark dispose d'un shell pour interpréter les instruction Python qui peut être invoqué avec la commande pyspark. Un objet SparContext est alors créé et placé dans la variable sc. Le programme est exécuté en simulant un nombre de nœuds égal au nombre de cores du logiques du processeur (voir Local[*] plus bas).

    Pour voir l'état du job, aller sur : http://localhost:4040. Cette interface n'est disponible que pendant pyspark est en cours d'exécution.

    Exécution des commandes

    Pour lancer les commandes shell utilisées dans les exemples :

    Lancer un terminal

    docker exec -it -u hadoop nodemaster bash

    Pour la suite les exemples sont relatives au cluster avec docker.

    Exemple Wordcount¶

    • Mettre le fichier shakespeare.txt sous HDFS.
    hadoop fs -put /opt/spark-apps/shakespeare.txt
    • Lancer pyspark
    pyspark
    • Exécuter les instructions suivantes une à une :
    texte = sc.textFile('shakespeare.txt')
count = texte.flatMap(lambda l:l.split()).map(lambda w:(w,1)).reduceByKey(lambda a,b:a+b)
count.collect()

    Spark UI¶

    Surveiller le job sur http://localhost:4040 remarquer que l'exécution ne se déclenche réellement qu'après l'appel de la méthode collect().

    Parmi les onglets de Spark UI :

    • Jobs : Un job est l’exécution d’une chaîne de traitements (workflow) dans un environnement distribué. Le workflow peut être visualisé à partir de DAG Visualization. Le nombre de jobs est éagal au nombre d'actions sur les RDDs. Dans cet exemple, l’exécution s’est faite en deux étapes (Stage). La première comprend les transformations textFile, flatMap et map, la seconde la transformation reduceByKey. les deux étapes sont séparées par une phase de shuffle. L’exécution d’une étape se fait par un ensemble de tâches, une par machine hébergeant un fragment du RDD.
    • Stages : Il montre de nombreuses statistiques sur le temps d’exécution, le volume des données échangées, ... de chaque étape.
    • Storage : Contient les données en mémoire pendant l'exécution. On peut demander explicitement de garder en mémoire un RDD avec cache() ou persist().
    persist()

    Ajouter les instructions suivantes :

    counts.persist()
counts.collect()

    Inspecter maintenant l'onglet Storage et comparer les temps d'exécution des deux instructions collect().

    spark-submit¶

    C'est un script Spark pour soumettre des jobs dans les langages acceptés sans interaction. Avec les paramètres il est possible de choisir le mode d'exécution, la réservation des ressources, la configuration Spark...

    La différence, par rapport à pyspark, est qu'il faut créer une instance SparkContext dans le programme Python soumis :

    sc = SparkContext(SparkConf(...))
    spark-submit
    spark-submit \
  --master <master-url> \
  --deploy-mode <client|cluster> \
  --conf <key<=<value> \
  --driver-memory <value>g \
  --executor-memory <value>g \
  --executor-cores <number of cores>  \
  --jars  <comma separated dependencies> \
  --py-files <comma separated dependencies> \
  --class <main-class> \
  <application-jar-py> \
  [application-arguments]

    En local¶

    Mode similaire à pyspark sans l'interactivité.

    spark-submit --master local[*] wordcount.py

    http://localhost:4040

    Spark standalone cluster¶

    spark-submit --master spark://nodemaster:7077

    http://localhost:8888

    yarn¶

    spark-submit --master yarn --deploy-mode cluster wordcount.py

    http://localhost:8080

    Surveillance avec Spark History Server¶

    Configuration spark-defaults.conf

    docker exec -it -u hadoop nodemaster bash
mkdir /tmp/spark-events
echo "spark.eventLog.enabled true" > spark/conf/spark-defaults.conf
echo spark.history.fs.logDirectory file:/tmp/spark-events

    Démarrer History Server

    docker exec -it -u hadoop nodemaster bash
./spark/sbin/start-historyserver.sh

    Accès à travers http://localhost:18080

    Soumettre des jobs avec spark-submit et les voir sur le history server.

    Activer History Server avec Yarn¶

    Pour activer la surveillance des application soumis à yarn :

    • Ajouter cette ligne à spark-defaults.conf (redémarrer historyserver) :
    org.apache.spark.deploy.yarn.YarnProxyRedirectFilter`à la liste de l'option `spark.ui.filters
    • Ajouter l'option --conf spark.yarn.historyServer.allowTracking=true à la commande spark-submit
    spark-submit --master yarn --conf spark.yarn.historyServer.allowTracking=true wordcount.py
